# Skrino: техническое описание

Нативное Windows-приложение на Rust. UI на egui/eframe 0.33 (glow/OpenGL), без webview и без рантаймов. Цели дизайна: мгновенная реакция на хоткей, минимум памяти в фоне, один самодостаточный exe.

## Структура workspace

```
crates/
  skrino-core     модель аннотаций, документ с undo/redo, экспорт-рендер
  skrino-capture  захват экрана (xcap), мониторы, склейка виртуального экрана
  skrino-upload   FTP/FTPS/SFTP, генерация имён, публичные ссылки
  skrino-record   запись видео (Windows.Graphics.Capture + Media Foundation, без ffmpeg)
  skrino-app      приложение: демон трея, оверлей, редактор, запись, настройки
```

Зависимости направлены строго вниз: `app -> {core, capture, upload, record}`; нижние крейты не знают ни про egui, ни друг про друга. Кроссплатформенность (план v2) сводится к замене реализации `capture`/`record` и системных мелочей в `app`.

## Двухпроцессная модель

Один exe, две роли:

- **`skrino --tray`** это фоновый демон: winit `EventLoop` без окна и без GPU (~9 МБ RAM). Держит иконку в трее (`tray-icon`) и два глобальных хоткея (`global-hotkey`); события приходят через колбэки в `EventLoopProxy`, в простое процесс не крутит циклов. Действия меню и хоткеи порождают отдельный UI-процесс (`current_exe()` + флаг) и не ждут его. Единственность экземпляра: named mutex `Global\skrino-tray`. Перечитывание конфига: демон держит поток, заблокированный на named event `Global\skrino-tray-reload`; процесс настроек сигналит его после сохранения, демон перерегистрирует хоткеи и обновляет подписи меню.
- **UI-процессы**: одноразовые `--capture-region`, `--capture-full`, `--record-region`, `--record-full`, `--open-file`, `--settings`, без флага (стартовое окно). Делают работу и завершаются вместе с окном; в фоне ничего не висит. Особый случай: если окно закрыли во время аплоада, процесс прячет окно и живёт до результата (жёсткий дедлайн 90 с), шлёт системное уведомление и умирает.

Хоткеи записи работают как переключатель: демон сначала пробует `OpenMutexW` по имени `Global\skrino-recording`; если запись уже идёт (мьютекс открылся), он шлёт `SetEvent` на named event `Global\skrino-record-stop` (активная запись останавливается), иначе порождает новый процесс записи. Пункты трея «Записать область»/«Записать весь экран» ведут себя так же. Флаг `--record-smoke` записывает primary-монитор 3 секунды без интерактивного UI, печатает путь MP4 и выходит: это единственный способ, которым запись проверяется автоматически.

## Оверлей выделения области (и почему именно так)

Исторический инцидент: первая реализация показывала оверлей отдельным дочерним viewport'ом, а корневое окно в это время пряталось. Windows перестаёт присылать redraw скрытому окну, цикл `update()` замирал, и на экране оставалась мёртвая полноэкранная always-on-top поверхность, съедавшая ввод; помогала только перезагрузка.

Текущая схема:

- захват выполняется **до** показа чего-либо (`capture_virtual_screen`), затем **корневое окно само** перешейпливается в безрамочный fullscreen always-on-top (`Decorations(false)`, `WindowLevel::AlwaysOnTop`, точные `OuterPosition`/`InnerSize`), и замороженный кадр с выделением рисуется в его `CentralPanel`;
- viewport-команды шлются **только на переходах состояний** (отслеживается «применённый» `WindowMode`), никогда каждый кадр;
- на время оверлея `request_repaint()` каждый кадр;
- предохранители: `Esc`, правый клик, авто-отмена через 120 с, и dev-флаг `--overlay-smoke` (автоотмена через 3 с и выход), единственный способ, которым оверлей тестируется автоматически.

Мультимонитор: оверлей накрывает монитор с курсором (`GetCursorPos` → `MonitorInfo`), из виртуального кадра вырезается срез этого монитора. Известное ограничение: на связках мониторов с разным DPI позиционирование использует масштаб целевого монитора и может ошибаться на пиксели.

## Запись экрана (skrino-record + control bar)

Движок записи (`skrino-record`) держит крошечный синхронный API: `is_supported()`, `Recorder::start/pause/resume/elapsed/take_error/stop/cancel`. Внутри Windows.Graphics.Capture отдаёт кадры аппаратному H.264-кодировщику Media Foundation, результат пишется прямо в MP4. Никаких внешних бинарников: ffmpeg не нужен, всё внутри `skrino.exe`. Требование: Windows 10 версии 1903 или новее (иначе `is_supported()` = false, и `app` показывает уведомление и выходит).

Ограничения текущей реализации: пишется только один монитор за раз (тот, где выбрана область; для полноэкранной записи это монитор под курсором), звук не записывается.

Оверлей выделения переиспользуется: у `OverlayState` есть параметр `OverlayPurpose` (`Screenshot`/`Record`), меняющий текст подсказки и то, что возвращает подтверждение (для скриншота обрезанный кадр, для записи прямоугольник `RegionPx` в физических пикселях виртуального экрана). Отдельного второго оверлея нет.

Порядок старта важен: полупрозрачный оверлей не должен попасть в видео. После подтверждения области корневое окно тем же процессом перешейпливается из fullscreen-оверлея в компактную панель управления (~224x48 логических px, `Decorations(false)`, малый `InnerSize`, позиция под областью, always-on-top, команды viewport только на переходе). Дальше идёт фаза «arming»: несколько кадров и ~180 мс на то, чтобы окно реально сжалось, и только потом стартует `Recorder`. За это время панель исключается из захвата вызовом `SetWindowDisplayAffinity(hwnd, WDA_EXCLUDEFROMCAPTURE)` (HWND берётся из `eframe::Frame` через `raw-window-handle`).

Панель управления: красная точка, таймер `mm:ss` (из `elapsed()`, `request_repaint_after(250 мс)`), пауза/продолжить, «Стоп» (акцент) и отмена; фон панели перетаскивается через `ViewportCommand::StartDrag` (только в момент начала перетаскивания). Позиция считается под нижней кромкой области, при нехватке места внизу уходит в правый верхний угол области, с клампом к монитору.

Рамка области: четыре тонких плоских слоёных Win32-окна (`record_frame.rs`, не egui-viewport'ы), `WS_EX_LAYERED|WS_EX_TRANSPARENT|WS_EX_TOOLWINDOW|WS_EX_TOPMOST|WS_EX_NOACTIVATE`, сквозные для кликов, сплошной жёлтый (`SetLayeredWindowAttributes`), тоже с `WDA_EXCLUDEFROMCAPTURE`. Создаются на старте, разрушаются на стопе/отмене/выходе (RAII-структура).

Запись идёт во временный файл (`%TEMP%\skrino-rec-<штамп>.mp4`), а «Стоп» пропускает его через ту же логику «Поделиться», что и скриншоты. На сервер: аплоад MP4 (`generate_filename("mp4")`, панель показывает «Загрузка…», ссылка в буфер, системный тост). Локально: перенос в папку сохранения с учётом «Спрашивать место» (rfd-диалог с фильтром `.mp4`). Отмена вызывает `cancel()` и тихо выходит; каждый кадр опрашивается `take_error()`, и при ошибке движка идёт системное уведомление и выход. Единственность: named mutex `Global\skrino-recording`, стоп-сигнал по auto-reset event `Global\skrino-record-stop` (поток-наблюдатель выставляет атомарный флаг и будит UI через сохранённый `egui::Context`).

## Координаты и DPI

`skrino-capture` работает в **физических пикселях** виртуального экрана (origin: левый верх primary-монитора, координаты бывают отрицательными). На Windows xcap 0.7 берёт геометрию из `EnumDisplaySettingsW` (уже физические пиксели) и `BitBlt`-ит ровно их, поэтому геттеры и буфер согласованы; на всякий случай размеры перепроверяются по фактическому буферу. UI (egui) живёт в логических пунктах; перевод выделения в пиксели картинки идёт через хелперы `transform.rs` (f32-математика, округление в самом конце), покрытые юнит-тестами, включая кейс 150% DPI.

## Модель редактора (skrino-core)

`Document` = базовый `RgbaImage` + `Vec<Annotation>` + неразрушающий `crop`. Аннотации **векторные** до момента экспорта (типы: Arrow, Line, Rect, Ellipse, Text, Marker, Pen, Blur, Counter). Undo/redo: стек снапшотов `(annotations, crop)`; база не мутируется никогда. Особенности:

- **Три стиля стрелки** (`ArrowHead`): Filled (сплошная + треугольный наконечник), Dashed (пунктирный стержень, тот же наконечник), Tapered (единый залитый контур со скруглениями, хвост 0.25·t расширяется до 0.8·t у шейки и 2·t у основания наконечника; наконечник ≤60% длины, на коротких стрелках все ширины масштабируются коэффициентом k, чтобы не было «блямбы»). UI-превью в редакторе повторяет формулы рендера. У Filled/Dashed стержень заканчивается за t·0.5 до основания наконечника (радиус круглого торца линии), чтобы торец не вылезал за края треугольника; половина ширины основания наконечника не меньше t·0.5+1 (`half_w = max(head_len·0.5, t·0.5+1)`), а на слишком короткой стрелке (короче head_len+t·0.5) стержень вовсе не рисуется, только голова. У Tapered хвост оформлен отдельным кружком радиуса tail_half (скруглённый колпачок), плечо у шейки и задние углы крыла наконечника скруглены фасками (радиус крыла ≈0.15·head_len), кончик чуть смягчён (0.3-1px), а грани стержня слегка вогнуты к оси (квадратичная кривая с контрольной точкой, смещённой на 12% к оси в середине). На экспорте (tiny-skia) это один путь с заливкой по правилу обхода, вогнутый угол у шейки ему не мешает; в редакторе (egui) `Shape::convex_polygon` веерит треугольники от первой вершины и не годится для невыпуклой фигуры, поэтому UI собирает тот же силуэт из выпуклых кусков (стержень, кружок-хвост, наконечник с фасками) - тот же вид, другая реализация.
- **Расширяемый холст**: `render::canvas_rect(doc)` = объединение прямоугольника базы и границ всех аннотаций (`annotation_bounds`, с учётом толщины/наконечников; текст: консервативная оценка без раскладки, UI обязан использовать ту же формулу). Аннотация за краем скриншота расширяет итоговую картинку, недостающее заливается **белым**. Blur холст не расширяет никогда (клампится к базе). Crop интерпретируется в этом же пространстве. Когда всё внутри базы, вывод байт-в-байт совпадает со старым поведением.

- **Blur**: единственная «растровая» аннотация, хранится как прямоугольник + sigma, применяется к копии базовых пикселей на рендере (3-проходный box blur ≈ гаусс) и всегда лежит **под** векторными фигурами независимо от порядка вставки. В UI предпросмотр это пикселизация (downscale + nearest upscale) с кэшем.
- **Экспорт-рендер** (`render_document`): tiny-skia (антиалиасинг, premultiplied alpha с явными конверсиями в обе стороны) + ab_glyph для текста. Порядок: blur по базе → векторные фигуры по порядку вставки → crop последним.
- **Шрифт один на всё**: Inter (OTF, кириллица) грузится и в egui, и в экспорт-рендер, поэтому текст на экране и в файле совпадает.

Группировка инструментов («Фигуры»: прямоугольник/эллипс/линия/метка; «Маркер»: маркер/карандаш) это чисто UI-слой поверх плоского `Tool` из core.

## Шрифты и иконки: две ловушки

1. **Inter маппит ~745 кодпоинтов в Private Use Area** и, стоя первым в фолбэк-цепочке, перекрывал часть иконок Phosphor (рендерились буквы вместо иконок).
2. **Стоковый Phosphor.ttf маппит пустые глифы на a–z** (триггеры лигатур) и, поставленный первым, съедал всю строчную латиницу.

Решение: в assets лежит `Phosphor-PUA.ttf`, сабсет только диапазона U+E000–F8FF (fontTools), он регистрируется первым; текст проваливается в Inter, иконки берутся из сабсета. Порядок шрифтов менять нельзя без понимания обеих ловушек.

## Аплоад (skrino-upload)

Синхронный публичный API (вызывать с worker-потока). FTP/FTPS: suppaftp (rustls; общая логика connect/login/mkdir/put раскатана макросом на оба типа стрима, т.к. трейт TLS-стрима приватный). SFTP: russh + russh-sftp поверх внутреннего однопоточного tokio-рантайма; авторизация паролем или ключом (`load_secret_key`, rsa-sha2 при поддержке сервером). Таймауты: connect 10 с, операция 60 с. Ошибки типизированы (`Connect/Auth/Transfer`) с человекочитаемыми сообщениями, UI показывает их как есть; на `Auth` тост предлагает открыть настройки.

Имена файлов: `YYYY-MM-DD_HH-MM-SS_<6 случайных a-z0-9>.<ext>`, сортируются по времени и не перебираются подбором. Публичная ссылка: шаблон с `{filename}`.

Известные упрощения (помечены TODO в коде): FTPS принимает любой сертификат, SFTP принимает любой host key (типичный кейс: свой сервер с self-signed). Пиннинг: кандидат на доработку.

## Конфиг и секреты

JSON в `%APPDATA%\skrino\config.json` (`serde`, `#[serde(default)]`, старые конфиги переживают новые поля). Пароль и парольная фраза ключа в JSON **не пишутся**: они хранятся только в Windows Credential Manager через `keyring` (сервис `skrino`); в конфиге лишь флаги `has_password`/`has_passphrase`. Окно настроек редактирует рабочую копию: на диск и в keychain всё попадает только по «Сохранить».

## Сборка и ресурсы

- `build.rs` + `winres` встраивают иконку exe из `assets/skrino.ico` (генератор из PNG: `examples/gen_ico.rs`).
- Release-профиль: `lto="thin"`, `codegen-units=1`, `strip`, `panic="abort"` → exe ~13 МБ.
- Криптография всюду на бэкенде `ring` (rustls и russh оба настроены на `ring`, aws-lc в дереве зависимостей нет), поэтому NASM для сборки не требуется.
- `#![windows_subsystem = "windows"]` в release: без консольного окна.

## Тесты

Около 108 юнит-тестов по всем крейтам: рендер каждой аннотации (включая кириллицу в тексте), blur/crop/границы, генерация имён и таймауты аплоада на закрытый порт, парсинг хоткеев (цифры, буквы, F-клавиши), сериализация конфига с обратной совместимостью, трансформации координат (включая 150% DPI), парсинг CLI-флагов, форматирование таймера и раскладка панели записи. Захват тестируется на живом железе (реальный монитор). Оверлей проверяется только через `--overlay-smoke`, запись только через `--record-smoke`.

## Планы v2

- Запись видео: готово (WGC + Media Foundation, MP4 без ffmpeg и без звука). Дальше: запись звука, выбор кодека и битрейта.
- Кроссплатформенность: реализация capture для macOS (ScreenCaptureKit + TCC) и Linux (X11/Wayland-порталы), системный слой app (трей, хоткеи, автостарт) за трейтами.
- Пиннинг TLS-сертификатов и SSH host keys.
